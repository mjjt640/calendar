# 软件需求规格说明书（SRS）

## 1. 引言

### 1.1 编写目的
本文档用于明确日程表 App 的功能需求、非功能需求、系统结构、数据要求和运行约束，作为项目设计、开发、测试与验收的依据。

### 1.2 项目背景
本项目拟开发一款基于 Android 原生 Java 的手机日程表应用，用于帮助用户进行个人日程安排、待办事项管理、提醒通知与数据备份恢复。系统强调本地优先、离线可用、操作流畅和结构清晰。

### 1.3 术语说明
- 日程：用户创建的计划事项
- 重复日程：按照周期规则自动重复生成的日程
- 提醒：在指定时间触发的本地通知
- 免打扰：在指定时间段限制普通提醒的策略
- 父线程：统一分配和调度后台任务的管理层
- 子线程：执行具体后台任务的线程或线程池工作线程

---

## 2. 总体描述

### 2.1 产品目标
开发一款适用于手机端的日程管理应用，实现高频、便捷、稳定的个人日程安排与提醒能力。

### 2.2 用户特征
目标用户包括：
- 学生
- 上班族
- 个人时间管理用户
- 课程设计或毕业设计使用者

### 2.3 运行环境
- 操作系统：Android
- 开发语言：Java
- 本地数据库：SQLite / Room
- 通知机制：系统本地通知

### 2.4 设计约束
- 以本地存储为主
- 核心功能必须支持离线运行
- 主线程不得执行耗时任务
- 提醒和后台处理必须使用统一线程调度策略

---

## 3. 功能性需求

## 3.1 日程管理

### 3.1.1 日程创建
系统应支持用户创建日程，并填写以下字段：
- 标题
- 详细描述
- 开始时间
- 结束时间
- 地点
- 重要程度（高 / 中 / 低）
- 标签分类

### 3.1.2 日程编辑
系统应支持对已创建日程进行完整编辑，包括修改标题、时间、地点、优先级、标签、提醒规则和重复规则。

### 3.1.3 日程删除
系统应支持：
- 单条删除
- 批量删除
- 删除时同步清理关联提醒

### 3.1.4 日程完成状态管理
系统应支持：
- 单条标记完成
- 批量标记完成
- 已完成状态的展示与筛选

### 3.1.5 日程查看
系统应支持：
- 日视图
- 周视图
- 月视图
- 月视图日期红点标记
- 点击日期查看当日日程列表

### 3.1.6 重复日程
系统应支持：
- 每日重复
- 每周重复
- 每月重复
- 自定义重复周期
- 单独修改某次重复日程实例
- 保留重复规则与例外记录

---

## 3.2 提醒功能

### 3.2.1 定时提醒
系统应支持为日程设置提醒时间，包括：
- 提前 5 分钟
- 提前 15 分钟
- 提前 30 分钟
- 提前 1 小时
- 自定义提醒时间

### 3.2.2 重复提醒
系统应支持未处理日程的间隔提醒，包括：
- 每 10 分钟提醒一次
- 每 30 分钟提醒一次
- 直到用户标记完成或手动关闭提醒

### 3.2.3 提醒方式
系统应支持：
- 声音提醒
- 弹窗提醒
- 用户自定义开关

### 3.2.4 免打扰模式
系统应支持：
- 用户设置免打扰时间段
- 免打扰期间抑制普通日程提醒
- 高优先级提醒可穿透免打扰模式

---

## 3.3 数据管理

### 3.3.1 本地存储
系统应将所有日程及配置默认存储在本地设备中，确保断网可正常使用。

### 3.3.2 数据备份
系统应支持：
- 导出 JSON 文件
- 导出 Excel 文件
- 备份文件保存到本地设备可访问目录

### 3.3.3 数据恢复
系统应支持从备份文件恢复数据，并在恢复前执行结构校验和异常处理。

### 3.3.4 数据清理
系统应支持：
- 一键清理已完成日程
- 一键清理过期日程
- 批量清理

---

## 3.4 界面交互

### 3.4.1 首页
首页应包含：
- 月视图日历
- 当日日程列表
- 快速新增入口

### 3.4.2 日程详情页
详情页应包含：
- 日程完整信息展示
- 快速编辑
- 删除
- 标记完成

### 3.4.3 添加日程页
添加页应包含：
- 简洁表单
- 关键信息快速填写
- 一键保存

### 3.4.4 设置页
设置页应包含：
- 提醒设置
- 主题设置
- 数据管理
- 账号管理
- 关于我们

### 3.4.5 主题切换
系统应支持：
- 浅色主题
- 深色主题
- 跟随系统主题自动切换

### 3.4.6 手势操作
系统应支持：
- 左滑删除
- 右滑标记完成
- 操作过程保持流畅

---

## 4. 非功能性需求

## 4.1 性能需求
- 首页首次加载时间应尽量控制在 2 秒内
- 月视图切换应平滑无明显卡顿
- 批量处理 100 条日程数据时应保持稳定响应
- 后台任务不得阻塞主线程

## 4.2 可用性需求
- 页面布局清晰，操作入口明确
- 表单录入步骤简洁
- 重要操作应提供视觉反馈和必要确认

## 4.3 可维护性需求
- 架构层次清晰
- 模块职责单一
- 线程调度集中管理
- 核心业务逻辑独立封装

## 4.4 可扩展性需求
后续应便于扩展以下能力：
- 云同步
- 账号系统
- 桌面组件
- 智能提醒
- 多设备同步

## 4.5 安全性需求
- 本地文件导入前需做格式校验
- 导入过程中需避免数据重复写入或主键冲突
- 清理和恢复操作需弹出确认提示

---

## 5. 系统架构需求

### 5.1 架构模式
系统采用 `MVVM + Repository + Room` 架构。

### 5.2 分层要求
- UI 层：负责界面与交互
- ViewModel 层：负责页面状态与业务调用
- Domain 层：负责业务规则与 UseCase
- Repository 层：负责数据访问统一出口
- Local Data Source 层：负责本地数据库与文件操作

### 5.3 并发模型
系统采用父子线程并发模型：
- 主线程负责界面显示和交互
- 父线程调度层负责统一提交和分发任务
- 子线程负责数据库、提醒、导入导出、规则生成等后台任务

### 5.4 并发约束
- 禁止在页面层直接创建线程
- 禁止在主线程执行数据库和文件读写
- 所有后台任务必须统一调度
- UI 更新必须回到主线程执行

## 7. 业务流程需求

### 7.1 创建日程流程
1. 用户填写表单并点击保存
2. 系统校验输入内容
3. 写入本地数据库
4. 生成提醒任务
5. 刷新首页与详情展示

### 7.2 重复日程流程
1. 用户设置重复规则
2. 系统保存规则
3. 后台线程生成重复实例
4. 若用户改单次实例，则记录为例外数据
5. 页面展示时优先应用例外实例

### 7.3 提醒流程
1. 到达提醒时间
2. 检查免打扰策略
3. 判断是否允许提醒
4. 发送本地通知
5. 若未处理，则按间隔重复提醒

### 7.4 数据恢复流程
1. 用户选择备份文件
2. 系统校验文件格式
3. 解析数据内容
4. 执行恢复写入
5. 刷新本地展示结果

---

## 8. 验收标准

### 8.1 功能验收
- 能正常新增、编辑、删除日程
- 能正常查看日 / 周 / 月视图
- 能正确显示红点与当日日程列表
- 能设置和执行提醒
- 能处理重复日程与例外修改
- 能导出和恢复数据
- 能切换主题与执行手势操作

### 8.2 性能验收
- 日常操作无明显卡顿
- 列表滚动流畅
- 后台任务执行时页面可正常交互

### 8.3 稳定性验收
- 无频繁闪退
- 异常输入可被正确拦截
- 导入导出失败时有明确提示
- 系统重启后提醒可恢复

## 10. 结论

本 SRS 文档用于指导日程表 App 的后续设计、编码、测试和验收。项目开发过程中应优先保证：
- 架构规范
- 主线程流畅
- 后台任务统一调度
- 数据可靠存储
- 提醒逻辑稳定可控
- 后续扩展方便
